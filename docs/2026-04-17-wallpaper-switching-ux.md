# 智能壁纸壁纸切换 UX 设计稿

## 1. 设计目标

本设计稿面向一个基于 React + Tauri 的桌面壁纸管理软件，目标不是做“图片浏览器”，而是让用户以尽可能低的认知负担完成以下核心任务：

1. 快速看懂当前每块屏幕正在使用什么壁纸
2. 在单屏、多屏、全屏统一三种模式下完成切换
3. 在切换前获得足够可信的预览反馈
4. 在失败、无权限、格式不支持、多显示器变化时明确知道系统发生了什么
5. 为未来的“自动切换 / 智能推荐 / 历史回滚”预留自然入口

本设计优先级如下：

- 第一优先级：切换动作可靠、可理解、可撤销
- 第二优先级：多显示器场景下不出错
- 第三优先级：让“选图到应用”的路径足够短
- 第四优先级：为后续自动化能力提供一致心智模型

## 2. 产品定义

### 2.1 核心用户场景

本阶段主要覆盖 4 类高频任务：

1. 我想立刻给当前主屏换一张壁纸
2. 我有多块显示器，想分别设置不同壁纸
3. 我想把同一张图同步到所有屏幕
4. 我希望软件按时间、规则或推荐结果自动切换

### 2.2 当前版本边界

基于现有后端能力，UX 设计默认支持：

- 枚举所有屏幕
- 读取当前壁纸状态
- 单屏设置
- 多屏分别设置
- 全屏统一设置
- 展示基础缩放模式
- 屏幕热插拔后的状态刷新
- 针对失败屏幕给出逐屏反馈

本稿不把以下内容纳入本期核心交互：

- 图片编辑器
- 云端图库
- 复杂批量标签管理
- 深度 AI 生成流程

这些能力可以在信息架构中预留入口，但不应影响首版壁纸切换主流程。

## 3. UX 设计原则

### 3.1 以“屏幕”为第一对象，不以“图片”为第一对象

用户真正要控制的是哪块屏幕显示哪张图，因此主页默认先展示屏幕状态，再让用户选图。这样在多显示器场景下最不容易出错。

### 3.2 所有切换都必须先有目标，再有动作

任何“应用壁纸”的动作都必须显式对应以下三类目标之一：

- 当前选中屏幕
- 所有屏幕
- 每个屏幕各自配置

避免出现用户选中图片后，不清楚会影响哪些屏幕的情况。

### 3.3 预览与最终应用分离

在桌面壁纸产品中，“看起来会怎样”与“真正写入系统”是两步。设计上要允许用户先试，再应用，减少误操作。

### 3.4 错误反馈必须指向具体屏幕

多显示器下，不能只提示“设置失败”。必须告诉用户：

- 哪块屏幕失败
- 为什么失败
- 其他屏幕是否已成功
- 接下来可做什么

### 3.5 自动切换不应打断手动切换

自动模式属于持续策略，手动切换属于即时任务。两者要并存，但层级上手动任务优先，避免用户误以为当前设置会被马上覆盖。

## 4. 信息架构

建议采用桌面端三栏布局。

```text
+----------------------------------------------------------------------------------+
| 顶栏: 当前工作模式 / 全局状态 / 刷新 / 自动切换开关 / 设置                         |
+----------------------+--------------------------------------+--------------------+
| 左栏                 | 中央                                  | 右栏               |
| 屏幕与模式           | 预览画布 + 当前配置                   | 图片源与操作面板   |
|                      |                                      |                    |
| - 全部屏幕           | - 按显示器比例展示预览                | - 最近使用         |
| - 内建显示器         | - 当前选中屏幕高亮                    | - 本地文件         |
| - 外接显示器 1       | - 应用结果反馈                        | - 收藏/推荐        |
| - 外接显示器 2       |                                      | - 缩放模式         |
|                      |                                      | - 应用按钮         |
+----------------------+--------------------------------------+--------------------+
```

### 4.1 左栏：屏幕导航区

左栏承担两个职责：

1. 告诉用户当前有哪些屏幕
2. 告诉用户当前切换模式是什么

建议结构：

- `全部屏幕`
- `逐屏设置`
- 屏幕列表

每个屏幕卡片显示：

- 屏幕名称，如“内建显示器”“DELL U2723QE”
- 主屏标签
- 分辨率或比例信息
- 当前壁纸缩略信息
- 最近一次状态：已应用 / 应用中 / 失败

### 4.2 中央：预览与结果区

中央区域是整个产品的“决策区”，必须最大、最直观。

包含 3 个层次：

1. 多屏布局预览
2. 当前选中屏幕的大图预览
3. 操作结果与差异提示

在“全部屏幕”模式下，中央主视图展示同一张图应用到所有屏幕后的效果。

在“逐屏设置”模式下，中央主视图展示多屏拼接布局，并高亮当前正在编辑的屏幕。

### 4.3 右栏：图片源与操作区

右栏用于完成“选图 + 调整 + 应用”。

建议从上到下分 4 个模块：

1. 图片来源切换
2. 图片列表/缩略图网格
3. 显示方式设置
4. 应用动作区

图片来源可以包括：

- 最近使用
- 本地文件夹
- 收藏
- 智能推荐

动作区按钮建议固定在底部，始终可见，避免滚动后找不到。

## 5. 关键交互模式

### 5.1 模式 A：快速切换当前屏幕

适合单屏用户或只想修改一块屏幕的场景。

流程：

1. 用户进入首页
2. 默认聚焦主屏
3. 右栏浏览图片
4. 点击某张图后，中央区域立即生成预览
5. 用户点击 `应用到当前屏幕`
6. 成功后显示完成反馈，并把该图写入最近使用

设计要点：

- 默认选中主屏，减少第一次使用成本
- 鼠标双击图片可触发“预览到当前屏幕”
- `应用` 按钮文案必须包含目标，如 `应用到内建显示器`

### 5.2 模式 B：统一应用到全部屏幕

适合用户想快速构建一致桌面风格。

流程：

1. 用户点击左栏 `全部屏幕`
2. 中央多屏预览进入统一模式
3. 右栏选择图片与缩放方式
4. 点击 `应用到全部屏幕`
5. 若存在失败，结果区展示成功/失败明细

设计要点：

- 统一模式下，所有屏幕外框都显示同一高亮色，表示编辑目标是全集
- `应用到全部屏幕` 使用强调按钮，但需在旁边明确显示受影响屏幕数量，如 `共 3 块`

### 5.3 模式 C：逐屏定制

这是多显示器用户的核心价值场景。

流程：

1. 用户进入 `逐屏设置`
2. 左栏展示所有屏幕
3. 点击某一屏幕后，中央预览高亮该屏
4. 右栏选择图片并调整缩放方式
5. 点击 `保存到该屏`
6. 用户继续切换到下一屏
7. 完成后点击顶部 `全部应用`

为避免用户在多屏配置中频繁等待，建议采用“草稿配置 + 批量应用”的模式。

具体机制：

- 用户每次选择图片后，先更新本地草稿状态
- 屏幕卡片显示 `未应用` 标识
- 顶部出现 `还有 2 处更改未应用`
- 用户统一点击 `全部应用`

这样比逐次直接写系统更高效，也更符合多屏配置心智。

### 5.4 模式 D：自动切换

自动切换不是单独页面，而是对现有切换模型的策略扩展。

建议入口放在顶栏右侧，使用一个清晰的全局开关：

- `自动切换：关闭`
- `自动切换：每小时`
- `自动切换：每天`

点击后打开抽屉或弹层，而不是跳转新页面。用户应在同一上下文中理解：

- 当前手动配置是什么
- 自动策略是否会覆盖它
- 下一次切换时间是什么

自动切换设置建议包含：

- 范围：当前屏幕 / 全部屏幕 / 逐屏独立
- 来源：收藏 / 某文件夹 / 推荐结果
- 频率：15 分钟 / 1 小时 / 每天 / 登录时
- 顺序：随机 / 最近未使用优先 / 固定列表轮播

## 6. 页面结构建议

### 6.1 顶栏

顶部信息从左到右建议为：

- 产品名或当前工作区名
- 当前模式标签
- 最近同步状态
- 刷新显示器按钮
- 自动切换开关
- 设置入口

状态文案示例：

- `已同步 3 块屏幕`
- `检测到显示器变化，已刷新`
- `2 处更改未应用`
- `自动切换将于 14:00 执行`

### 6.2 左栏屏幕卡片

屏幕卡片建议结构如下：

```text
+-----------------------------------+
| 内建显示器          主屏          |
| 3024 x 1964                       |
| 当前: mountain-snow.jpg           |
| 状态: 已应用                      |
+-----------------------------------+
```

交互规则：

- 单击：选中该屏幕
- 右键或更多菜单：复制设置、粘贴设置、设为统一目标
- 拖拽排序：仅改变 UI 展示顺序，不改变系统屏幕编号

### 6.3 中央预览画布

中央预览区要模拟真实多屏相对布局，而不是等宽平铺。

设计要求：

- 按真实宽高比缩放显示
- 主屏带轻微强调描边
- 选中屏幕显示更强高亮
- 鼠标悬停某个屏幕时同步高亮左栏卡片
- 如果图片与屏幕比例差异大，预览中展示裁剪区域

预览区顶部可有一个简短的上下文标题：

- `正在编辑：内建显示器`
- `正在编辑：全部屏幕`

### 6.4 右栏操作区

右栏建议结构：

```text
图片来源 Tabs
- 最近使用
- 本地图库
- 收藏
- 推荐

图片网格
- 4 列缩略图
- 悬停显示尺寸和文件名
- 当前选中图显示描边与勾选

显示设置
- 填充
- 适应
- 拉伸
- 居中
- 平铺

动作区
- 预览
- 应用到当前屏幕
- 应用到全部屏幕
```

规则：

- 不同模式下只显示一个主按钮，避免主次混乱
- 次级动作用文本按钮，如 `恢复当前系统壁纸`

## 7. 关键任务流

### 7.1 首次打开

首次打开目标是建立信任，而不是迫使用户立刻配置。

流程建议：

1. 自动读取所有屏幕和当前壁纸
2. 中央显示当前桌面实际状态
3. 如果无本地图库，右栏提供 `导入文件夹` 和 `选择图片`
4. 如果存在权限问题，在顶部显示非阻断提示条

首屏不应出现空白大页面。哪怕用户还未导入图库，也应先展示系统当前壁纸状态。

### 7.2 选择图片并预览

选择图片后的响应要足够快，最好在 100-200ms 内完成本地预览反馈。

状态变化：

1. 缩略图进入选中态
2. 中央预览替换为候选图片
3. 右栏动作区激活
4. 若图片过小或比例不适合，显示轻量提示

提示示例：

- `该图片分辨率低于当前屏幕，应用后可能模糊`
- `该图片将按“填充”模式裁剪`

### 7.3 应用成功

成功反馈要轻，但必须明确。

建议：

- 在按钮上显示短暂 loading
- 成功后在顶栏出现 2-3 秒状态文案
- 对应屏幕卡片状态更新为 `已应用`

文案示例：

- `已应用到内建显示器`
- `已应用到全部 3 块屏幕`

### 7.4 部分失败

多屏场景中部分失败是必须单独设计的。

结果提示区示例：

```text
已完成 2/3 块屏幕
失败: DELL U2723QE
原因: 图片无访问权限
[重试] [更换图片] [查看详情]
```

规则：

- 成功的屏幕立即更新状态
- 失败屏幕保留错误标记
- 不回滚已经成功的屏幕
- 提供逐屏重试入口

### 7.5 屏幕热插拔

如果显示器数量变化，系统应自动刷新，但不要粗暴打断用户。

建议交互：

- 顶部出现提示：`检测到显示器变化`
- 中央预览平滑重排
- 若用户正在编辑已移除的屏幕草稿，则将其移动到 `待处理变更` 区域并提示用户确认

## 8. 状态设计

### 8.1 全局状态

建议前端显式维护以下全局状态：

- `idle`
- `loading_snapshot`
- `editing_draft`
- `applying`
- `partial_failure`
- `success`
- `display_changed`

### 8.2 屏幕级状态

每块屏幕卡片建议有独立状态：

- `synced`
- `draft`
- `applying`
- `applied`
- `failed`
- `disconnected`

这样能避免一个全局 loading 把全部屏幕都锁死。

### 8.3 图片项状态

图片缩略图建议具备：

- `default`
- `hover`
- `selected`
- `previewing`
- `unsupported`
- `missing`

## 9. 文案策略

桌面工具的文案要直接，不使用模糊表述。

推荐文案风格：

- `应用到当前屏幕`
- `应用到全部屏幕`
- `还有 2 处更改未应用`
- `该图片格式暂不支持`
- `无法访问该图片，请检查文件权限`
- `已检测到新的显示器`

避免：

- `确认`
- `提交`
- `执行`
- `操作成功`

这些词没有告诉用户具体发生了什么。

## 10. 视觉方向建议

虽然本稿不是高保真视觉稿，但建议明确视觉方向，避免做成通用文件管理器。

### 10.1 整体气质

建议采用“工作台式”桌面视觉：

- 深色石墨灰或暖灰作为基础底色
- 局部用偏冷青绿色作为交互强调色
- 大量使用真实缩略图与屏幕布局作为视觉主体
- 控件边框克制，减少传统后台感

### 10.2 视觉层级

层级顺序建议为：

1. 中央预览画布
2. 当前选中屏幕
3. 主按钮
4. 左右栏辅助信息

不要让右侧图片网格视觉上压过中央预览，否则产品会被误解为图库应用。

### 10.3 动效建议

动效只服务于理解状态变化：

- 切换选中屏幕时，中央高亮框平滑过渡
- 应用成功后，对应屏幕边框做一次轻微闪动
- 屏幕布局变化时使用位置过渡，不要瞬间跳变

动效时长建议控制在 160ms 到 240ms。

## 11. React + Tauri 落地建议

### 11.1 前端状态分层

建议拆成三层状态：

1. 系统快照状态
2. 本地编辑草稿状态
3. 应用任务状态

对应心智如下：

- 系统快照：来自 Tauri 后端的真实状态
- 草稿状态：用户正在尝试但尚未写入系统的配置
- 应用任务状态：某次提交过程中的 loading / success / failure

这样可以自然支持“先编辑多个屏幕，再统一应用”。

### 11.2 推荐组件拆分

```text
WallpaperWorkspace
|- TopbarStatus
|- ScreenNavigator
|- DisplayCanvasPreview
|- ImageSourceTabs
|- WallpaperGrid
|- WallpaperOptionPanel
|- ApplyActionBar
|- AutoSwitchDrawer
|- ResultToast / InlineResultPanel
```

### 11.3 Tauri 事件协作

建议 Rust/Tauri 层提供：

- 获取当前屏幕与壁纸快照
- 应用单屏壁纸
- 应用全部屏幕壁纸
- 应用逐屏批量设置
- 订阅显示器变更事件

前端收到显示器变更事件后：

1. 标记当前快照为 stale
2. 触发静默刷新
3. 若本地有未应用草稿，提示用户确认合并方式

### 11.4 错误模型映射

后端已有逐屏错误模型，因此前端不要只做 toast。

建议错误 UI 分两层：

- 全局 toast：告诉用户这次操作整体成功还是部分失败
- 局部卡片错误：标记具体哪块屏幕失败

## 12. 首版 MVP 建议

如果当前目标是尽快启动前端实现，建议按以下优先级收敛：

### P0

- 屏幕列表
- 中央多屏预览
- 本地图片选择
- 单屏应用
- 全屏应用
- 成功/失败反馈

### P1

- 逐屏草稿模式
- 批量应用
- 最近使用
- 缩放模式切换
- 屏幕热插拔提示

### P2

- 自动切换抽屉
- 收藏
- 推荐源
- 历史回滚

## 13. 设计结论

本产品的壁纸切换 UX 不应以“找图片”为中心，而应以“管理屏幕状态”为中心。首页以屏幕布局和当前状态建立上下文，右侧图片选择承担素材输入，中央预览承担结果确认，顶部状态承担系统反馈。这样既适合单屏快速切换，也能平滑支持多屏定制与未来自动切换能力。

对于 React + Tauri 的实现，建议优先落地“三栏工作台 + 草稿/应用分层状态 + 逐屏错误反馈”这三个基础结构。这会直接决定首版是否稳定、清晰、可扩展。
